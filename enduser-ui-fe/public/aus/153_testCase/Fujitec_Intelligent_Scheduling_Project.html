Part I: Foundational Data Engineering

The "Garbage In, Garbage Out" Principle

The most fundamental principle in any data analytics project is that the quality of the output is entirely dependent on the quality of the input. Fujitec's current data environment presents a classic 'messy data' challenge, fragmented across disparate sources. The primary objective of this part is to conquer this chaos by constructing a robust Extract, Transform, and Load (ETL) pipeline. This pipeline will ingest the raw data and methodically cleanse, standardize, and integrate it into a single, reliable dataset known as an Analytic Base Table (ABT).

Chapter 2: Normalizing and Ingesting Raw Data

2.1. Why Normalize to a Standard Format?

Fujitec's data is delivered in a mix of Excel (`.xlsx`) and CSV formats. Using `PROC IMPORT` with `DBMS=XLSX` allows SAS to directly read modern Excel files, avoiding potential data corruption issues from manual CSV conversions and creating a more reliable ingestion pipeline.

2.2. A Detailed `PROC IMPORT` Walkthrough

The first program, `01_ingest_data.sas`, reads the raw files into a dedicated "staging" library, isolating the raw ingested data for more organized and auditable processing.

/*======================================================================
  PROGRAM: 01_ingest_data.sas
  PURPOSE: Ingest raw Excel files into staging SAS datasets.
======================================================================*/

/* A LIBNAME statement assigns a nickname to a physical directory for easier management. */
libname staging '/projects/fujitec/working_data/staging';

/* --- Ingest Work Reporting Data --- */
PROC IMPORT 
    /* DATAFILE= points to the exact location of the source Excel file. */
    DATAFILE="/projects/fujitec/raw_data/Fujitec_Work_Reporting.xlsx" 
    /* OUT= specifies the new SAS dataset to be created (libname.dataset_name). */
    OUT=staging.work_reporting_raw 
    /* DBMS=XLSX tells SAS this is a modern Excel file. This is crucial. */
    DBMS=XLSX 
    /* REPLACE allows this code to be re-run without failing. Useful for development. */
    REPLACE;
    
    /* SHEET= specifies which worksheet to read. Must match the name exactly. */
    SHEET="Work Reporting Data";
    
    /* GETNAMES=YES tells SAS the first row contains column headers for variable names. */
    GETNAMES=YES;
RUN;

/* --- Ingest PLM/Process Design Data --- */
PROC IMPORT 
    DATAFILE="/projects/fujitec/raw_data/Fujitec_PLM_Process.xlsx" 
    OUT=staging.plm_process_raw 
    DBMS=XLSX 
    REPLACE;
    SHEET="PLM Data";
    GETNAMES=YES;
RUN;

/* --- Ingest ERP System Data --- */
PROC IMPORT 
    DATAFILE="/projects/fujitec/raw_data/Fujitec_ERP_Data.xlsx" 
    OUT=staging.erp_raw 
    DBMS=XLSX 
    REPLACE;
    SHEET="ERP Data";
    GETNAMES=YES;
RUN;

Chapter 3: The Transformation Pipeline - A Deep Dive into Data Cleansing

With raw data staged, we now clean it. The core strategy is to isolate cleansing logic from ingestion logic. For each raw staging table, a corresponding cleaned table will be created. This modular approach greatly simplifies debugging and auditing.

3.1. Handling Inconsistent Text and IDs

Inconsistent text is a common point of failure. For known variations (e.g., "J.Smith" vs "jsmith"), we'll use `PROC FORMAT` to create a reusable standardization map.

Pro Tip: Use PROC FORMAT

`PROC FORMAT` is superior to long `IF-THEN/ELSE` chains because it centralizes the standardization logic in a reusable "map," making it easier to read, update, and maintain.

/* 1. Define the standardization format. It acts as a lookup table. */
PROC FORMAT;
  /* The VALUE statement defines the mapping for the format named $EmpID_Std. */
  VALUE $EmpID_Std
    /* List of input values on the left, standardized output value on the right. */
    'J.SMITH', 'JSMITH', 'SMITH.J' = 'EMP_12345'
    'T.JONES', 'JONES.T' = 'EMP_67890'
    /* The OTHER keyword assigns a default for any value not explicitly listed. */
    OTHER = 'UNKNOWN_ID';
RUN;

/* 2. Apply the format in a DATA step */
DATA work.work_reporting_clean_step1;
  SET staging.work_reporting_raw;
  
  /* Apply the custom format using the PUT() function to create a new, standardized variable */
  Standard_Employee_ID = PUT(UPCASE(TRIM(Employee_ID)), $EmpID_Std.);
RUN;

3.2. The Date Conundrum: Standardizing Multiple Date Formats

Handling inconsistent date formats is a classic data engineering challenge. The most reliable method is to read the date as a character string, then use conditional logic with the `INPUT()` function and the flexible `ANYDTDTE.` informat to convert it to a standard SAS date value (a number representing days since Jan 1, 1960).

DATA work.work_reporting_clean_step2;
  SET work.work_reporting_clean_step1;
  
  /* LENGTH 8 is standard for numeric variables. FORMAT DATE9. displays it as DDMMMYYYY. */
  LENGTH Standard_Work_Date 8.;
  FORMAT Standard_Work_Date DATE9.;

  /* The ANYDTDTE. informat is very powerful; it tries to automatically detect
     many common date formats that use delimiters like /, -, or space. */
  Standard_Work_Date = INPUT(Work_Date_Raw, ANYDTDTE40.); 
  
  /* If conversion fails, the value will be missing. We can flag this for review. */
  IF missing(Standard_Work_Date) THEN PUT "WARNING: Unrecognized date format for " Work_Date_Raw;

  DROP Work_Date_Raw;
  RENAME Standard_Work_Date = Work_Date;
RUN;

3.3. Cleansing Numeric Fields and Removing Duplicates

Numeric fields can contain errors like non-numeric text or impossible values (e.g., negative hours). These must be cleaned. After standardizing key fields, `PROC SORT` with the `NODUPKEY` option efficiently removes duplicate records.

/* This final step in the cleaning process cleans numerics and removes duplicates. */
DATA work.work_reporting_clean_tmp;
    SET work.work_reporting_clean_step2;
    /* The ?? modifier on the informat tells SAS not to write errors to the log if conversion fails. */
    Actual_Man_Hours_Num = INPUT(Actual_Man_Hours_Raw, ??8.);
    /* Set invalid values to missing, which SAS handles correctly in calculations. */
    IF Actual_Man_Hours_Num < 0 THEN Actual_Man_Hours_Num = .;
    DROP Actual_Man_Hours_Raw;
    RENAME Actual_Man_Hours_Num = Actual_Man_Hours;
RUN;

PROC SORT DATA=work.work_reporting_clean_tmp OUT=work.work_reporting_clean NODUPKEY;
  /* The BY statement lists all variables that define a unique record.
     NODUPKEY removes any observation where this combination has already been seen. */
  BY Standard_Employee_ID Work_Date Work_Order_No;
RUN;

Chapter 4: Final Assembly - Creating the Analytic Base Table (ABT)

The ABT is the "single source of truth" for the project. It's a single, wide table that joins all our cleaned data sources. This design is optimal for modeling and visualization as it eliminates the need for complex joins later on. The creation of the ABT directly addresses Fujitec's core problem of "information silos."

/*======================================================================
  PROGRAM: 03_create_abt.sas
  PURPOSE: Joins cleaned tables into the final Analytic Base Table (ABT).
======================================================================*/

/* Define a permanent library for the final ABT. */
libname prod '/projects/fujitec/working_data/production';

PROC SQL;
  CREATE TABLE prod.fujitec_abt AS
  SELECT
    t1.Work_Order_No,
    t1.Operation_Code,
    t1.Work_Date,
    t1.Actual_Man_Hours,
    t1.Standard_Employee_ID AS Employee_ID,
    t2.Required_Skill,
    t3.Due_Date,
    t3.Material_ID
  FROM
    work.work_reporting_clean AS t1
    /* LEFT JOIN ensures that all records from the primary table (work reporting)
       are kept, even if there isn't a match in the other tables. */
    LEFT JOIN work.plm_process_clean AS t2
      /* The ON clause specifies the key(s) used to match records. */
      ON t1.Operation_Code = t2.Operation_Code
    LEFT JOIN work.erp_clean AS t3
      ON t1.Work_Order_No = t3.Work_Order_No;
QUIT;

Part II: The Optimization Core: Building the Scheduling Engine

With a reliable ABT, we can now build the analytical "brain" of the solution: an intelligent scheduling engine using `PROC OPTMODEL`. The goal is to translate Fujitec's complex scheduling problem into a mathematical model that finds a provably optimal production schedule.

Chapter 5: A Practitioner's Guide to Optimization Techniques

5.1. Why Mixed-Integer Linear Programming (MILP)?

We've chosen MILP because it finds the best possible outcome (e.g., minimum completion time) and, most importantly, provides a mathematical **guarantee of optimality**. This rigor is essential to prove that our intelligent system is superior to manual scheduling. While other techniques might find "good" solutions faster, they cannot provide this same level of certainty, which is essential for making a compelling business case to Fujitec.

Chapter 6: Translating "Tribal Knowledge" into Mathematical Code

6.1. The Power of `PROC OPTMODEL`

A huge risk in scheduling projects is failing to capture the unwritten rules used by experienced planners. `PROC OPTMODEL`'s syntax closely mimics mathematical algebra, making it easy to translate a business rule (e.g., "Task A must finish before Task B can start") directly into a transparent, auditable line of code. This process transforms implicit, person-dependent knowledge into an explicit, robust corporate asset.

6.2. Setting up the Model

We'll create a new program, `04_optimize_schedule.sas`. The model follows a logical progression: declare sets and parameters, read data, declare decision variables, define constraints, set the objective, solve, and export.

Chapter 7: Formulating the Constraints - The Rules of the Factory

Constraints are the mathematical statements that define the rules the schedule must obey. They are the heart of the model.

7.1. Precedence Constraints

Business Rule: A successor task cannot begin until its predecessor is complete. SAS Code: We translate this directly. For a task `t` that must follow task `p`, the logic is: `StartTime[t] >= StartTime[p] + TaskDuration[p]`.

/* --- CONSTRAINT: PRECEDENCE --- */
/* This ensures that for each task 't' with a predecessor 'p', the start time of 't'
   must be greater than or equal to the start time of 'p' plus its duration. */
CON Precedence_con {t IN TASKS, p IN PREDECESSORS[t]}:
  StartTime[t] >= StartTime[p] + TaskDuration[p];

7.2. The Critical Workforce Constraints

Business Rule: Every task must be assigned to exactly one employee who has the required skill. This directly models and solves Fujitec's single-skilled workforce bottleneck.

/* --- CONSTRAINT: SKILL REQUIREMENT & SINGLE ASSIGNMENT --- */
/* For each task 't', this ensures exactly one employee is assigned.
   The summation is performed ONLY over the set of employees 'e' who possess
   the skill required for task 't' (where HasSkill[e, RequiredSkill[t]] = 1). */
CON SkillRequirement_con {t IN TASKS}:
  SUM{e IN EMPLOYEES: HasSkill[e, RequiredSkill[t]] = 1} Assign[e, t] = 1;

Chapter 8: Solving the Model and Exporting the Optimal Schedule

8.1. Defining the Objective: Minimize Makespan

The business goal is to minimize the makespan (the completion time of the last job). We define a `Makespan` variable and tell the solver to make it as small as possible.

/* --- CONSTRAINT: MAKESPAN DEFINITION --- */
/* This defines the Makespan. It must be >= the completion time of every task. */
CON Makespan_def_con {t IN TASKS}:
  Makespan >= StartTime[t] + TaskDuration[t];

/* --- OBJECTIVE FUNCTION --- */
/* The objective is to minimize the Makespan. The solver will adjust decision
   variables (StartTime, Assign) to find a valid schedule with the smallest
   possible value for the Makespan variable. */
MIN Objective = Makespan;

8.2. Invoking the Solver and Exporting the Solution

A single `SOLVE` statement triggers the powerful MILP solver. After a successful solve, we use `CREATE DATA` to extract the optimal values (e.g., `StartTime[t].sol`) and save them to a new SAS dataset, ready for the dashboard.

/* This statement tells SAS to find the guaranteed optimal solution. */
SOLVE WITH MILP;

/* --- EXPORT THE SOLUTION --- */
/* The CREATE DATA statement extracts the optimal solution values and saves them.
   This table is the primary output of the optimization engine. */
CREATE DATA prod.optimized_schedule FROM [t]=TASKS
  COLUMNS Task_ID=t
          /* The .sol suffix retrieves the optimal value found by the solver. */
          Task_Start_Time=StartTime[t].sol 
          Task_End_Time=(StartTime[t].sol + TaskDuration[t])
          /* This finds the employee 'e' for whom the binary Assign variable is 1. */
          Assigned_Employee=({e IN EMPLOYEES: Assign[e,t].sol > 0.5} e)
          Machine=TaskMachine[t];

Part III: The Decision-Support Layer: From Data to Decisions

The raw output from the optimization is just numbers. This layer translates this into clear, actionable intelligence using an interactive dashboard in SAS Visual Analytics. This combats the "limited visibility" of Fujitec's current Excel process with an intuitive, "at-a-glance" interface.

Chapter 9: Building the Dashboard in SAS Visual Analytics

9.1. Building the Gantt Chart

The Gantt chart is the visual centerpiece. We can build a highly effective one using a standard Bar Chart object in SAS VA.

Gantt Chart Trick

Use a stacked horizontal bar chart. The first measure is the task's start time (`Gantt_Start`), which you make transparent. The second measure is the task's duration (`Gantt_Duration`), which you color. This creates the classic "floating bar" effect of a Gantt chart.

Chapter 10: Empowering Managers with "What-If" Scenarios

A cornerstone of this POC is providing managers a tool to proactively manage disruptions. This moves them from reactive firefighting to proactive contingency planning. We achieve this using Parameters in SAS Visual Analytics.

  1. Generate Scenario Data: In SAS Studio, run the optimization model multiple times with different assumptions (e.g., "Baseline", "Welder Absent"). Combine the results into one dataset with a `Scenario` column.
  2. Create a Parameter in VA: Create a new character parameter named `Scenario_Selection`.
  3. Create a User Control: Add a Drop-down list to the report, linking it to the `Scenario` column and the `Scenario_Selection` parameter.
  4. Link to Visuals: Filter the report page to only show data where the `Scenario` column equals the `Scenario_Selection` parameter. When a user selects a scenario, the entire dashboard updates instantly.

Part IV: Intelligent Automation: Connecting Analytics to Action

An optimal plan is only valuable if it's integrated into the workflow. This final part addresses the "last mile" of analytics by demonstrating how to connect the intelligence from SAS to Fujitec's downstream systems, providing a scalable blueprint for integrating with legacy platforms.

Chapter 11: The "Brain and Hands" - Intelligent Process Automation (IPA)

We'll combine advanced analytics with Robotic Process Automation (RPA) in a "Brain and Hands" model:

  • The Brain (SAS Analytics for IoT (SAS VIYA)): Performs the complex decision-making to create the optimal schedule.
  • The Hands (RPA Bot): Executes simple, repetitive tasks based on the brain's decision. For this POC, the bot will take the schedule and automatically populate a standard Excel report.

Chapter 12: The Integration Workflow

The integration is a three-step process that uses `PROC HTTP` to make a REST API call to the RPA platform's endpoint, sending a JSON payload to trigger the bot.

/* This PROC HTTP step sends the JSON payload to the RPA platform's API endpoint. */
PROC HTTP
  /* The URL is the specific API endpoint provided by the RPA platform. */
  URL="<RPA_ORCHESTRATOR_API_URL>"
  /* METHOD="POST" is used to send data to a server. */
  METHOD="POST"
  /* IN= specifies the fileref that contains the request body (our JSON payload). */
  IN=json_payload_fileref;
  HEADERS
    /* This header tells the server that the data being sent is in JSON format. */
    "Content-Type"="application/json"
    /* Most modern APIs require an authentication token for security. */
    "Authorization"="Bearer <RPA_PLATFORM_API_TOKEN>";
RUN;

The bot receives the API call, parses the JSON, opens an Excel template, and populates it with the schedule data. This completes the end-to-end intelligent automation workflow.
